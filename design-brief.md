# Pi Agent Web — 功能需求文档

> 本文档只描述：用户流程、每个节点的功能需求、系统需要展示什么、用户可以操作什么。
> 不指定布局、不画线框、不限制交互形式。设计师根据需求自行决定界面结构。


## 产品概述

Pi Agent Web 是 Pi Coding Agent 的浏览器界面。用户跟 AI 对话、让它自动读代码/跑命令/改文件、必要时手动接管和调教。

与普通聊天 AI 的关键区别：它不只是回答问题，而是真正执行操作。用户需要知道 Agent 在干什么、改了哪些文件、什么时候需要自己介入。

系统还有一套 **Multi-Agent 协作** 机制：当一个任务比较复杂时，系统会自动（或用户手动）启用多个子 Agent 并行工作，由一个 Lead Agent 统一规划和审查。这部分是高级功能，但对重度用户是核心价值。


## 系统角色说明

用户需要感知到以下角色（设计师不需要把每个角色做成独立界面，但需要知道这些角色的含义，以便在状态展示中准确表达）：

| 角色 | 说明 | 用户需要感知的程度 |
|------|------|-------------------|
| **主模型** | 用户日常对话的 AI | 一直可见 |
| **Guardian** | 便宜模型的"安全员"，判断任务是否复杂到需要启用 Multi-Agent | 只在 Multi-Agent 相关状态中看到 |
| **Lead Agent** | Multi-Agent 的主控者，负责任务规划、审查子 Agent 产出、做最终汇报 | Multi-Agent 模式下可见 |
| **Sub-Agent** | 执行具体子任务的 Agent，可以有不同的 Profile（长期角色） | Multi-Agent 模式下可见 |
| **Agent Coach** | Lead 调教链路中的强模型示范角色，不是独立角色 | 仅在"调教"流程中出现 |
| **Artifact Reviewer** | 审稿助手，帮 Lead 审查子 Agent 产出物 | 仅在审查阶段出现 |

## 基础功能（始终可用的界面元素）

以下功能不属于某个特定流程，而是用户在应用中的任何时候都可能使用到的常驻能力。

### 文件浏览器（Explore）

**用户场景**：需要在对话的同时浏览项目文件结构、打开文件查看内容、将文件引用到对话中。

**与 Agent 对话的关系**：独立的浏览行为，不打断对话。用户可以先浏览文件再决定问什么，也可以边聊天边看文件。

**系统需要展示**：
- 当前项目目录的完整文件树（可折叠层级）
- 文件图标（根据类型区分）
- 当前打开的文件内容（代码语法高亮、Markdown 渲染）
- 支持同时打开多个文件，以 Tab 形式切换

**系统需要响应**：
- 展开/折叠文件夹
- 点击文件 → 在预览区打开文件内容
- 引用按钮（@） → 将文件路径插入到对话输入框
- 关闭文件 Tab

---

### 对话分支（Branches）

**用户场景**：对话不是线性的。用户可以在某条消息处分叉出一个新分支——要么在当前对话内回头重来，要么创建一个完全独立的新会话。

**两种分支操作**：

| 操作 | 效果 |
|------|------|
| **Edit from here** | 在当前会话内，从选中的消息重新开始。这条消息之后的内容被覆盖，形成分支节点。用户可以在不同分支间跳转。 |
| **Fork 新会话** | 从选中的消息创建完全独立的会话副本。两个会话互不影响。 |

**系统需要展示**：
- 当前会话的分支结构（树状）
- 当前所在分支位置
- 分支节点的消息摘要
- 不同分支上的消息内容

**系统需要响应**：
- 在用户消息上 hover → 显示 Edit from here / Fork 按钮
- 切换分支 → 加载对应分支的消息
- 分支间的导航和跳转

---

### System Prompt 查看

**用户场景**：高级用户需要了解当前会话使用的系统提示词（决定了 Agent 的行为规则）。

**系统需要展示**：
- 当前生效的 System Prompt 全文（Markdown 格式）
- 如果有自定义替换或追加，显示来源

**设计要点**：
- 这是低频功能，但高级用户需要时能快速找到
- 不需要持续占据屏幕空间
- 有自定义 System Prompt 时，可以用小标记提示用户

---

### 暗色 / 浅色模式切换

**系统需要展示**：当前主题状态（暗色/浅色）。

**系统需要响应**：点击切换，过渡有平滑动画。

---

### 模型切换

**用户场景**：对话过程中随时可以更换 AI 模型。不同模型有不同的能力、速度、价格。

**系统需要展示**：
- 当前使用的模型名称（简短显示，如 "gpt-5.5"）
- 所有可用模型列表，按 provider（模型供应商）分组
- 每个模型的连接状态（可用 / 未接入）
- 支持搜索模型名称

**系统需要响应**：
- 切换模型 → 后续对话使用新模型
- 显示/隐藏未接入的模型
- 新对话时，自动选择默认模型

**模型切换与 Agent 的关系**：
- 切换模型只影响后续发送的消息
- Agent 正在运行时也可以切换（但当前消息不受影响）
- 如果开启了自动模型选择，新会话会使用系统推荐的默认模型

---

### 工具模式切换

**用户场景**：控制 Agent 能调用哪些内置工具。不同场景需要不同的工具权限。

| 模式 | 可用工具 | 适合场景 |
|------|---------|---------|
| 关闭 (off) | 无工具 | 纯聊天、问概念 |
| 基础 (default) | read, bash, edit, write | 日常编程任务 |
| 全部 (full) | 基础 + grep, find, ls | 需要搜索/遍历项目 |

**设计要点**：
- 基础模式已经包含 bash 和 edit（能跑命令、能改文件），所以基础到全部的差别不大
- 真正有意义的选择是：关闭（纯聊天）vs 有工具（能执行操作）

---

### 推理强度切换

**用户场景**：控制模型在生成回复前的思考深度。

| 级别 | 含义 |
|------|------|
| 自动 | 沿用系统默认 |
| 关闭 | 不额外推理 |
| 最低 ~ 最高 | 分 5 档推理强度 |

**设计要点**：
- 不是所有模型都支持所有推理级别
- 前端需要显示当前模型实际可用/不可用的级别
- 不同模型对同一级别的实际行为可能不同（需要显示映射后的实际值）

---

### 声音提示

Agent 执行完成时可以选择播放提示音。用户可以开关。

---

### Models 配置管理

**用户场景**：管理模型供应商的 API 配置（API key、endpoint 等）。

**系统需要展示**：已配置的模型列表、各自的连接状态、配置详情。

**系统需要响应**：编辑配置、测试连接。

---

### Skills 管理

**用户场景**：管理 Agent 的技能模块（Skills），如 "frontend-design"、"pdf"、"tavily-search" 等。

**系统需要展示**：
- 已安装的 Skills 列表
- 每个 Skill 的启用/禁用状态
- Skill 功能描述

**系统需要响应**：
- 开关 Skill（启用/禁用）
- 删除已安装的 Skill
- 搜索并安装新 Skill

**重要区分**：
- Skills 面板管理的是 Pi 普通技能资源
- Multi-Agent 的 Profile 系统是另一层（Agent 长期画像），不在 Skills 面板管理
- Agent Workbench 中的 Skills 开关是"本次任务级别的调整"，不影响 Skills 面板的全局开关

---

### 上下文压缩

**用户场景**：对话太长时，上下文窗口会快满，Agent 可能忘记早期内容。用户可以手动压缩，也可以让系统自动触发。

**系统需要展示**：
- 当前上下文使用率（百分比 + 总容量）
- 使用率 > 70% 时视觉提示（黄色）
- 使用率 > 90% 时醒目提示（红色）
- 压缩中显示进度
- 压缩完成后显示效果（压缩前/后 token 数对比）

**系统需要响应**：
- 手动触发压缩
- 压缩过程中可以取消
- 自动压缩触发时也显示同样反馈

---

### Token 用量与费用统计

**用户场景**：了解当前会话的 token 消耗和费用。

**系统需要展示**：
- 输入 token 数
- 输出 token 数
- cache 读取 token 数
- cache 写入 token 数
- 累计费用（美元）

**设计要点**：这是辅助信息，只需要轻量展示，不要抢占主要注意力。用户可以 hover 查看详情。

## 一、主对话流程

### 1.1 进入应用

**用户需要做什么**：选择一个项目文件夹开始对话，或者打开历史对话。

**系统需要展示**：
- 产品名称
- 选项目入口：拖入文件夹 / 手动输入路径 / 最近用过的项目列表
- 历史对话列表（如果存在）

**系统需要响应**：
- 拖入文件夹 → 验证路径有效性 → 进入新对话
- 点击最近项目 → 进入新对话
- 点击历史对话 → 加载该对话的全部消息

---

### 1.2 正常对话

**用户需要做什么**：输入消息 → 看回复 → 继续追问。偶尔引用文件到输入框。

**系统需要展示**：

用户消息：
- 文字内容（可能含图片附件）
- 发送时间

Agent 回复：
- 文字内容（支持 Markdown、代码高亮、数学公式、流程图）
- 当前使用的模型名称
- 累计 token 用量和费用
- 完成时间

Agent 使用的工具（默认折叠为一行摘要）：
- 全部成功时："已使用 6 个工具 · 32s"
- 有文件修改时，额外显示："已修改 3 个文件 — AppShell.tsx, ChatInput.tsx, MessageView.tsx"
- 有文件新建："已新建 1 个文件 — new-component.tsx"
- 有文件删除："已删除 1 个文件 — old-file.tsx"
- 有错误："1 个工具执行失败 — npm test 返回错误码"
- 工具摘要为可展开项，展开后显示每个工具的调用参数和执行输出

**系统需要响应**：
- 发消息 → Agent 开始工作
- 点击工具摘要 → 展开工具列表
- 点击文件修改摘要 → 展开文件列表
- 复制回复文本
- 拖拽图片 → 作为附件发送
- 从文件列表引用 → 输入框插入文件路径

**特殊输入方式**：
- 拖拽图片到对话区 → 自动附加
- 点击文件 @ 按钮 → 输入框插入文件路径

---

### 1.3 Agent 正在运行时

**用户需要做什么**：看进度、中途补指令、必要时停止。

**系统需要展示**：
- Agent 运行中标识
- 当前正在执行的操作（如"正在读取 AppShell.tsx"、"正在执行 npm test"）
- 已完成步骤数/总步骤数（如果有）

**接管操作**：
- **立即插入**：打断 Agent 当前操作，把用户消息注入给它（主操作按钮）
- **完成后发送**：等 Agent 完成当前工作后再处理这条消息（次操作，下拉或次级按钮）
- **停止**：立即终止 Agent

**为什么需要两种发送方式**：
- 立即插入：用户发现 Agent 理解错了，要马上纠正
- 完成后发送：用户知道 Agent 现在忙，只是想追加一点要求，不用打断

---

## 二、Agent 修改文件后

**用户需要做什么**：查看 Agent 改了什么，确认或要求修改。

**系统需要展示**：
- 修改/新建/删除的文件列表
- 每个文件的操作类型（修改/新建/删除）和状态（成功/失败）
- 点击后可以预览文件内容（或 diff）

**用户表达不满意时**：
系统检测到"不满意/不对/继续调教/重新优化"等关键词后，触发 **Coach 调教链路**（见 4.3）。

## 三、Multi-Agent 协作

Multi-Agent 是本产品的核心差异化功能。当任务比较复杂时，Guardian（便宜模型）自动判断并启用——或者用户手动开启——系统会启动一个 Lead Agent 做规划，然后派发多个子 Agent 并行执行。

### 3.1 Multi-Agent 模式控制

**三种模式**：
1. **自动模式**（默认）：Guardian 自动判断任务复杂度，决定是否启用
2. **手动开启**：用户主动要求启用
3. **本会话关闭**：用户手动关过后，Guardian 不再自动启用，表示用户接管控制权

**用户需要看到**：
- 当前处于哪种模式
- 如果 Guardian 自动启用了 Multi-Agent，有一个明确的状态变化
- 如果用户手动关过，显示"用户接管"标记

### 3.2 Multi-Agent 运行时的全局状态

**Session 状态（整个 Multi-Agent 任务的阶段）**：

| 阶段 | 含义 | 用户看到什么 |
|------|------|------------|
| guardian | Guardian 正在分析任务复杂度 | "正在分析任务..." |
| planned | Lead 已完成任务规划 | "已规划 N 个子任务" |
| running | 子 Agent 正在执行 | 显示执行中状态 |
| waiting_confirmation | 有子 Agent 需要用户确认才能继续 | 醒目的确认提示 |
| synthesizing | Lead 正在审查子 Agent 产出 | "正在审查结果..." |
| done | 全部完成，Lead 给出最终汇报 | 最终汇报内容 |
| error | 出错了 | 错误信息 |

### 3.3 子 Agent（Task）状态

每个子 Agent 在整个生命周期中会经历以下状态：

| 状态 | 含义 |
|------|------|
| pending | 等待执行（还没轮到，或等待依赖的其他 Agent 完成） |
| waiting_for_dependency | 等待上游 Agent 的产出物 |
| waiting_confirmation | 需要用户确认才能继续 |
| running | 正在执行 |
| completed | 执行完成，产出物已生成，等待 Lead 审查 |
| paused | 用户手动暂停 |
| error | 执行出错 |
| aborted | 用户或系统终止 |

### 3.4 子 Agent 的协作状态

除了基础状态，每个子 Agent 还有协作协议层面的状态：

| 协作状态 | 含义 |
|---------|------|
| debugging | 子 Agent 初次执行此类型任务，进入调试模式（Lead 会更密切监控） |
| waiting_material | 等待上游交付物 |
| waiting_agent_decision | 等待其他 Agent 的决策 |
| waiting_lead_decision | 等待 Lead 的判断 |
| waiting_user_confirmation | 等待用户确认 |
| ready_for_review | 产出完成，等待 Lead 审查 |
| needs_revision | Lead 审查认为需要修订 |
| accepted | Lead 审查通过，接受产出 |
| blocked | 阻塞，无法继续 |

### 3.5 用户需要看到的 Multi-Agent 信息

**整体层面**：
- 当前 Session 处于哪个阶段（guardian / planned / running / synthesizing / done）
- 有几个子 Agent、分别处于什么状态
- 哪些子 Agent 需要用户确认
- 最终汇报内容

**每个子 Agent 层面**：
- 名称（如"前端审查"、"后端修改"）
- 当前状态和协作状态
- 使用的模型
- 依赖关系（这个 Agent 依赖谁、被谁依赖）
- 如果出错，错误原因
- 这个子 Agent 所属的 Profile（如果有的话）

**产出物（Artifact）层面**：
- 每个子 Agent 产出了什么
- 产出物状态：ready（可用）/ incomplete（不完整）/ needs_revision（需要修订）
- 如果有质量问题，什么问题（如"输出格式不符合要求"）
- 产出物被哪些下游 Agent 消费了

### 3.6 用户能做的 Multi-Agent 操作

- 查看整体阶段流程
- 点击某个子 Agent → 进入 Agent 工作台（见 4.1）
- 暂停 / 恢复 / 终止某个子 Agent
- 给某个子 Agent 切换模型
- **用户确认**：当子 Agent 需要确认时，用户可以选择"确认继续"或"停止"

## 四、Agent 调教与 Profile 系统

这部分是产品最深的差异化：用户不只是用一次 Agent，而是可以长期培养它。

### 4.1 Agent 工作台

**入口**：在 Multi-Agent 协作面板中点击某个子 Agent。

**用户需要看到**：
- Agent 基本信息：名称、当前模型、状态
- 它所属的 Profile（长期角色）信息：
  - Profile 名称
  - Profile 默认模型
  - Profile 固定技能
  - Profile 可选技能池
  - Profile 的项目配置（如偏好文件路径）
  - Profile 的历史成功/失败次数
  - Profile 最近保存的经验
- 本次任务信息：
  - 原始 prompt（Lead 给这个 Agent 的任务描述）
  - 当前输出或错误信息
  - 任务阶段列表
  - 是否需要方案讨论（Lead 认为还需要讨论方案，不是直接执行）
  - Lead 的下一步建议

**用户能做的操作**：

1. **调整本次 Skills**：点亮或取消某个 skill，影响本次重跑
2. **写补充指令**：只给这个 Agent 看的本次指令（不改全局 prompt，不改 profile）
3. **重跑**：让 Agent 按新的 skills 和指令重新执行
4. **切换模型**：给这个 Agent 换一个模型执行

**关于 Skills 的说明**：
- Skills 是 Agent 的能力模块（如"frontend-design"、"pdf"、"tavily-search"等）
- 本次激活的 skills 由 Lead 根据任务需要自动选择
- 用户可以在工作台手动调整，本次生效
- Skills 的长期归属权在 Profile 里

### 4.2 模型路由：为什么某个 Agent 用了这个模型

每个子 Agent 使用的模型，背后有决策逻辑。用户需要看到：

| 模型来源 | 含义 |
|---------|------|
| Lead 选择 | Lead 根据任务类型主动指定 |
| Profile 默认 | 该 Agent 的 Profile 预设模型 |
| 用户覆盖 | 用户手动切换的 |
| 固定路由 | 系统规则强制指定 |
| 保护性改派 | 系统发现任务涉及关键文件（如 lead-agent.md），自动换到更安全的模型 |

用户需要看到：**实际使用的是什么模型、为什么用这个模型**。

### 4.3 Coach 调教链路

**触发条件**：
- 用户表达不满意（"这里不对"、"继续调教"、"改得不对"等）
- 或在 Agent 工作台中主动发起

**调教流程**：

1. Lead 判断：用户反馈是否需要进入调教链路
   - 不需要：返回原因（如"反馈不够具体"）
   - 需要：进入下一步

2. Lead 生成"理想方案"
   - 由强模型（Lead 自己）输出一个正确版本的示范
   - 同时分析子 Agent 之前为什么做不好

3. 子 Agent 重做
   - 拿到 Lead 的理想方案和纠偏指导
   - 重新执行，尽量贴近理想方案

4. Lead 对比判断
   - 新输出是否足够接近理想方案
   - 如果不够 → 继续迭代
   - 如果够 → 调教结束

5. 调教结束后：
   - 生成最终方案（Lead 强化过的）
   - 生成 teaching note（"这个 Agent 以后应该注意什么"）
   - 如果该 Agent 有 Profile，teaching note 自动写入 Profile 的经验沉淀

**用户需要看到**：
- 当前是否处于调教链路中
- 调教了几轮
- Lead 的理想方案是什么
- 子 Agent 的新输出跟理想方案有多接近
- 最终的 teaching note 是什么
- 如果写入 Profile，需要明确显示"已保存到 Profile X"

### 4.4 保存经验到 Profile

**这是什么**：用户确认某个子 Agent 这次的做事方式以后应该长期保持。

**操作**：
- 在 Agent 工作台中展开"保存经验"
- 写一段 lesson（如"这个 Agent 擅长处理前端组件重构，推荐激活 frontend-design skill"）
- 保存

**保存后系统会做什么**（用户需要知道）：
- 这次使用的 skills 合并到 Profile 的固定技能池
- lesson 追加到 Profile 的 system prompt（系统提示词补丁）
- 这次的任务记录加入 Profile 的 savedExperiences 列表

**设计要点**：
- 这不只是"记笔记"，而是**直接修改 Profile 的长期行为**。用户需要明确知道这一点。
- 不要让人误以为只是保存一个备忘。

### 4.5 升级为新 Profile

**这是什么**：用户认为某个子 Agent 的表现值得成为一个独立的、可复用的 Agent 模板。

**跟"保存经验"的区别**：
- 保存经验 → 在已有 Profile 上追加一条经验
- 升级为 Profile → 创建一个全新的 Profile

**操作流程**：
1. 在 Agent 工作台中展开"升级为新 Profile"
2. 输入 Profile 名称（如"前端组件重构 Agent"）
3. 输入描述（这个 Profile 什么时候用）
4. 确认升级

**系统会生成什么**（用户需要看到）：
- 新 Profile 的名称和 ID
- 继承的 skills
- 默认模型
- 来源：是 Coach 确认后升级，还是用户直接确认升级
- 当前是"候选 profile"（待多次验证后才能成为正式 profile）

**设计要点**：
- 区分"候选 profile"和"正式 profile"
- 升级操作需要谨慎，不要让用户误触。但不妨碍流畅操作。
- 显示新 Profile 的关键信息，但不一次性堆太多。

### 4.6 审查策略

Lead 审查子 Agent 产出时，有不同策略：

| 策略 | 含义 |
|------|------|
| lead_plus_reviewer | 默认，Lead 审查 + Artifact Reviewer 辅助审稿 |
| lead_only | 只 Lead 审查，跳过 Artifact Reviewer（用户要求直出时） |

**用户需要知道**：
- 当前审查策略是什么
- 如果用户说了"直接给我结果/不用审查"，系统会跳过额外的审稿环节
- 如果用户说了"仔细审查/严格把关"，系统会启用双重审查

**未来方向**（尚未完全落地）：
- 成熟 Profile 的产出可以免审直出
- 用户是最终质量判断者
- 如果结果不好，走调教链路纠偏

## 五、Artifact 系统（物料/交付物）

### 5.1 什么是 Artifact

每个子 Agent 执行完成后会产生一个交付物（Artifact）。它不只是 Agent 的输出文字，而是一个有格式、有状态、可被下游消费的资产。

### 5.2 Artifact 信息

| 属性 | 说明 |
|------|------|
| 名称/ID | 唯一标识 |
| 类型 | 报告 / 代码 / 规划 / 分析 |
| 状态 | ready（可用）/ incomplete（不完整）/ needs_revision（需修订） |
| 生产者 | 哪个子 Agent 产出的 |
| 消费者 | 哪些下游 Agent 会用到它 |
| 质量问题 | 如有，具体问题描述 |
| 版本 | 版本号 |
| 摘要 | 内容摘要 |

### 5.3 Artifact 质量评估

子 Agent 执行完成后，系统会自动评估产出质量：
- 如果输出格式不符合要求 → 标记为 incomplete，要求 Agent 修正一次
- 如果修正后仍不合格 → 仍然保存，但标记质量问题，交给 Lead 审查时处理

### 5.4 用户需要看到的 Artifact 信息

- 当前有哪些 Artifact
- 每个 Artifact 的状态（ready / incomplete）
- 每个 Artifact 的生产者和消费者
- 如果有质量问题，什么问题

## 六、项目记忆

Multi-Agent 系统会维护项目级别的记忆，跨对话持久化。

### 6.1 记忆内容

- **Context**：当前项目背景、目标
- **Progress**：当前进展、已完成的事项
- **Bugs**：已知 bug 和待修复项
- **Recent Summaries**：最近几次会话的摘要

### 6.2 用户操作

- 查看项目记忆
- 刷新记忆
- 清除摘要

## 八、核心体验原则

以下不是功能需求，而是体验目标。设计师自行判断如何达成：

1. **主对话区是核心**。用户大部分时间在看对话，不要让其他信息挤小它。

2. **工具执行是后台行为**。Agent 读文件、跑命令的细节默认折叠，用户想看才展开。

3. **文件修改要醒目**。Agent 可能静默改了很多文件，用户需要一眼看到改了什么、改了几个。

4. **运行中的 Agent 可控**。用户可以随时补指令、中止、查看进度。这些操作要直接、不犹豫。

5. **Multi-Agent 按需可见**。不启用时完全不干扰主界面，启用时提供清晰的并行任务视图。

6. **长期行为调整要谨慎**。保存经验、升级 Profile 这些操作应让用户明确知道影响的是长期行为，不是一次性的。

7. **状态要能区分紧急程度**。正常运行 / 需要用户关注 / 需要用户确认 / 有错误，四个层级有明显视觉差异。

8. **模型路由透明**。用户可以在需要时查到 Agent 为什么用这个模型。

9. **常驻功能轻量化**。文件浏览、模型切换、分支管理、主题切换这些常驻功能是高频操作，但不应抢占对话主区域。

10. **熟悉感优先**。大部分时间用户只是在聊天。日常状态安静、熟悉。高级功能按需出现。
